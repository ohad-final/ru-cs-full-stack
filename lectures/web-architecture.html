Monolith vs Microservices

Monolith
Single Deployment Unit
Auth Users Orders Payments Notifications
Single Database
  • Simple to develop & deploy
  • Easy debugging (one codebase)
  • No network latency between modules
  • ACID transactions are trivial
  • Scales as a unit (wasteful)
  • One bad deploy affects everything
  • Tech stack locked in
VS
Microservices
Auth
Users
Orders
Payments
DB DB DB DB
  • Independent scaling
  • Team autonomy
  • Technology diversity
  • Fault isolation
  • Network complexity
  • Distributed transactions are hard
  • Operational overhead (K8s, service mesh)

The "Majestic Monolith"

Most startups should start here. Microservices are earned, not chosen.

Shopify $175B GMV, still a monolith
Basecamp Millions of users, 3M LOC monolith
Stack Overflow 50M visitors/month, 9 servers

When to Consider Microservices

✓ Multiple teams (10+) need to deploy independently
✓ Wildly different scaling requirements per feature
✓ Clear domain boundaries already exist
✗ "It's what Netflix does" (you're not Netflix)
✗ "Microservices are modern" (complexity is not modern)

Part II

API Architecture

REST, GraphQL, gRPC, and When to Use Each

REST API Best Practices

1. Resource Naming

/getUsers, /createUser, /user/delete/123 GET /users, POST /users, DELETE /users/123

Nouns for resources, HTTP verbs for actions

2. Nested Resources

/users/123/orders — Orders belonging to user 123 /orders/456/items — Items in order 456

Max 2 levels deep. Beyond that, use query params.

3. Pagination & Filtering

/users?page=2&limit=20&sort=created_at&filter[status]=active

4. Meaningful Status Codes

200 OK 201 Created 204 No Content 400 Bad Request 401 Unauthorized 403 Forbidden 404 Not Found 429 Rate Limited

Richardson Maturity Model

How "RESTful" is your API?

Level 0
The Swamp of POX
Single endpoint, POST everything
POST /api — { "action": "getUser", "id": 123 }
Level 1
Resources
Individual endpoints per resource
POST /users/123 — { "action": "get" }
Level 2
HTTP Verbs
GET, POST, PUT, DELETE properly used
GET /users/123
Level 3
HATEOAS
Hypermedia controls in responses
{ "user": {...}, "_links": { "orders": "/users/123/orders" } }
Reality check: Most APIs are Level 2. HATEOAS is rarely worth the complexity.

GraphQL — Query What You Need

GraphQL Query
query {
  user(id: "123") {
    name
    email
    orders(last: 5) {
      id
      total
      items {
        name
        price
      }
    }
  }
}

Why GraphQL?

  • No over-fetching (get only what you need)
  • No under-fetching (one request, all data)
  • Strongly typed schema
  • Great for mobile (bandwidth matters)

GraphQL Challenges

N+1 Problem

Fetching 100 users with orders = 101 DB queries without DataLoader

Caching Complexity

No HTTP caching — every query is POST

Complexity Attacks

Deeply nested queries can DDoS your server

Use GraphQL when:
  • Multiple clients need different data shapes
  • Rapid frontend iteration
  • Complex, nested data relationships

gRPC & Protocol Buffers

When JSON is too slow and you control both ends

Protocol Buffer Definition

syntax = "proto3";

message User {
  int32 id = 1;
  string name = 2;
  string email = 3;
}

service UserService {
  rpc GetUser(GetUserRequest) returns (User);
  rpc StreamUsers(Empty) returns (stream User);
}
10x smaller

Binary format vs JSON text

Type-safe

Generated client/server code

Streaming

Bidirectional streaming built-in

HTTP/2

Multiplexing, header compression

Use gRPC for: Internal microservice communication, real-time systems, mobile (low bandwidth)

Webhooks & Event APIs

Polling (Pull)

Client → GET /status → Server

"Are we there yet? Are we there yet?"

  • Wastes resources
  • Delayed updates

Webhooks (Push)

Server → POST /callback → Client

"I'll call you when it's ready"

  • Real-time updates
  • Efficient

Webhook Best Practices

Idempotency

Handle duplicate deliveries gracefully

Retry with Backoff

1s → 5s → 30s → 5min → give up

Signature Verification

HMAC-SHA256 to verify sender

Respond Fast

Return 200 immediately, process async

API Versioning Strategies

How to evolve your API without breaking clients

URL Path Versioning

/v1/users, /v2/users
Easy to understand & cache Not "pure" REST
Used by: Twitter, Stripe, GitHub

Header Versioning

Accept: application/vnd.api+json;version=2
Clean URLs Harder to test in browser
Used by: GitHub (also), Azure

Query Parameter

/users?version=2
Easy to add Caching complications
Used by: Google, Amazon
Recommendation: URL path (/v1/) is clearest. Avoid breaking changes — add fields, deprecate old ones.

Full Stack Patterns

Clean Architecture, DDD, and Real-World Examples

Clean Architecture

Frameworks & Drivers Express, React, Postgres
Interface Adapters Controllers, Presenters, Gateways
Use Cases Application business rules
Entities Enterprise business rules
The Dependency Rule: Dependencies point INWARD. Inner circles know nothing about outer circles.
Benefit: Swap your database, framework, or UI without touching business logic.

CQRS & Event Sourcing

Command Side (Write)

CreateOrder → Command Handler → Write DB
|

Query Side (Read)

GetOrders → Query Handler → Read DB

Event Sourcing

Store events, not state. Rebuild state by replaying events.

OrderCreated → ItemAdded → PaymentReceived → OrderShipped
When to use: High read/write ratio, audit requirements, complex domains. Overkill for most apps.

Domain-Driven Design Basics

Bounded Contexts

Same word, different meaning in different contexts

"Customer" in Sales ≠ "Customer" in Support

Aggregates

Cluster of objects treated as a unit

Order (Root) → OrderItems → ShippingAddress

Ubiquitous Language

Developers and domain experts use the same terms

Code mirrors business language
Start simple: You don't need DDD for a todo app. Use it when domain complexity is the problem.

Architecture Decision Checklist

Before Choosing

  • [ ] What are our scale requirements (now vs 2 years)?
  • [ ] How many developers/teams will work on this?
  • [ ] What's our deployment frequency?
  • [ ] What existing systems must we integrate?
  • [ ] What's our team's expertise?

Warning Signs

  • [ ] "It's what [Big Tech] uses"
  • [ ] "It's the modern way"
  • [ ] "We might need it someday"
  • [ ] Architecture diagram is bigger than your team
  • [ ] More services than developers
"The best architecture is the one you can evolve from."

Key Takeaways

01
Start with a monolith

Split when you have clear boundaries and team scale demands it.

02
Cache aggressively

Browser → CDN → App → DB. The fastest request is one you don't make.

03
REST is usually enough

GraphQL for mobile/complex UIs, gRPC for internal services.

04
Complexity is not a feature

Simple architectures are easier to understand, debug, and evolve.
