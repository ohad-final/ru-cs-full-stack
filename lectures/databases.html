ACID Properties

The guarantees that make relational databases reliable

A

Atomicity

All or nothing. Transaction either completes fully or rolls back entirely.

Transfer $100: debit AND credit both succeed or both fail
C

Consistency

Database moves from one valid state to another. Rules are always enforced.

Balance can never go negative if constraint exists
I

Isolation

Concurrent transactions don't interfere with each other.

Two users buying last item: only one succeeds
D

Durability

Once committed, data survives crashes and power failures.

Confirmed order stays confirmed even if server dies

SQL — The Language of Data

DDL — Define Structure
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  name VARCHAR(100) NOT NULL,
  email VARCHAR(255) UNIQUE,
  created_at TIMESTAMP DEFAULT NOW()
);

ALTER TABLE users ADD COLUMN role VARCHAR(50);
DROP TABLE IF EXISTS old_users;
DML — Manipulate Data
INSERT INTO users (name, email)
VALUES ('Alice', 'alice@example.com');

UPDATE users SET role = 'admin' WHERE id = 1;

DELETE FROM users WHERE created_at < '2020-01-01';
DQL — Query Data
SELECT u.name, t.name as team
FROM users u
JOIN teams t ON u.team_id = t.id
WHERE u.role = 'engineer'
ORDER BY u.created_at DESC
LIMIT 10;

PostgreSQL with Node.js

Using pg (node-postgres)
import { Pool } from 'pg';

const pool = new Pool({
  connectionString: process.env.DATABASE_URL
});

// Query with parameterized values (safe!)
async function getUser(id: number) {
  const result = await pool.query(
    'SELECT * FROM users WHERE id = $1',
    [id]
  );
  return result.rows[0];
}

// Transaction example
async function transfer(from: number, to: number, amount: number) {
  const client = await pool.connect();
  try {
    await client.query('BEGIN');
    await client.query(
      'UPDATE accounts SET balance = balance - $1 WHERE id = $2',
      [amount, from]
    );
    await client.query(
      'UPDATE accounts SET balance = balance + $1 WHERE id = $2',
      [amount, to]
    );
    await client.query('COMMIT');
  } catch (e) {
    await client.query('ROLLBACK');
    throw e;
  } finally {
    client.release();
  }
}

When to Use Relational Databases

Perfect For

  • Financial systems requiring ACID
  • Complex relationships between entities
  • Ad-hoc reporting and analytics
  • Data integrity is critical
  • Well-defined, structured schema
PostgreSQL MySQL SQLite

Consider Alternatives When

  • Schema changes frequently
  • Horizontal scaling is priority
  • Data is hierarchical/nested
  • Ultra-low latency required
  • Working with unstructured data
💡

PostgreSQL handles 10,000+ writes/second on modest hardware. Don't assume you need NoSQL for "scale".

MongoDB with Node.js

import { MongoClient, ObjectId } from 'mongodb';

const client = new MongoClient(process.env.MONGODB_URI);
const db = client.db('myapp');
const users = db.collection('users');

// Insert a document
await users.insertOne({
  name: 'Bob Smith',
  email: 'bob@example.com',
  createdAt: new Date()
});

// Find with query operators
const activeDevs = await users.find({
  'profile.skills': { $in: ['TypeScript', 'React'] },
  lastLogin: { $gte: new Date('2024-01-01') }
}).toArray();

// Update nested fields
await users.updateOne(
  { _id: new ObjectId('507f1f77bcf86cd799439011') },
  {
    $push: { 'profile.skills': 'GraphQL' },
    $set: { 'profile.level': 'senior' }
  }
);

// Aggregation pipeline
const postStats = await users.aggregate([
  { $unwind: '$posts' },
  { $group: { _id: '$name', totalLikes: { $sum: '$posts.likes' } } },
  { $sort: { totalLikes: -1 } }
]).toArray();

Redis with Node.js

import Redis from 'ioredis';

const redis = new Redis(process.env.REDIS_URL);

// Basic key-value
await redis.set('user:1:name', 'Alice');
await redis.get('user:1:name'); // "Alice"

// With expiration (great for caching)
await redis.setex('session:abc123', 3600, JSON.stringify({ userId: 1 }));

// Hash - perfect for objects
await redis.hset('user:1', { name: 'Alice', email: 'alice@example.com' });
await redis.hgetall('user:1'); // { name: 'Alice', email: '...' }

// Lists - for queues, feeds
await redis.lpush('notifications:1', 'New message!');
await redis.lrange('notifications:1', 0, 9); // Get last 10

// Sorted Sets - leaderboards, rankings
await redis.zadd('leaderboard', 100, 'alice', 85, 'bob', 92, 'carol');
await redis.zrevrange('leaderboard', 0, 2, 'WITHSCORES');
// [['alice', '100'], ['carol', '92'], ['bob', '85']]

// Pub/Sub for real-time
redis.subscribe('chat:room1');
redis.on('message', (channel, message) => {
  console.log(`${channel}: ${message}`);
});

Cypher — Graph Query Language

// Create nodes and relationships
CREATE (alice:Person {name: 'Alice', role: 'Developer'})
CREATE (bob:Person {name: 'Bob', role: 'Designer'})
CREATE (ts:Skill {name: 'TypeScript'})
CREATE (alice)-[:KNOWS]->(ts)
CREATE (bob)-[:KNOWS]->(ts)
CREATE (alice)-[:FRIENDS_WITH]->(bob)

// Find friends of friends
MATCH (me:Person {name: 'Alice'})-[:FRIENDS_WITH]->
      (friend)-[:FRIENDS_WITH]->(foaf)
WHERE foaf <> me
RETURN DISTINCT foaf.name

// Shortest path between two people
MATCH path = shortestPath(
  (a:Person {name: 'Alice'})-[*]-(b:Person {name: 'David'})
)
RETURN path

// Recommendation: skills my friends know that I don't
MATCH (me:Person {name: 'Alice'})-[:FRIENDS_WITH]->(friend)-[:KNOWS]->(skill)
WHERE NOT (me)-[:KNOWS]->(skill)
RETURN skill.name, COUNT(friend) as friendsWhoKnow
ORDER BY friendsWhoKnow DESC

Prisma — The Modern ORM

schema.prisma
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

model User {
  id        Int      @id @default(autoincrement())
  email     String   @unique
  name      String?
  posts     Post[]
  profile   Profile?
  createdAt DateTime @default(now())
}

model Post {
  id        Int      @id @default(autoincrement())
  title     String
  content   String?
  published Boolean  @default(false)
  author    User     @relation(fields: [authorId], references: [id])
  authorId  Int
}

model Profile {
  id     Int    @id @default(autoincrement())
  bio    String
  user   User   @relation(fields: [userId], references: [id])
  userId Int    @unique
}

Prisma Query Examples

import { PrismaClient } from '@prisma/client';

const prisma = new PrismaClient();

// Create with relations
const user = await prisma.user.create({
  data: {
    email: 'alice@example.com',
    name: 'Alice',
    posts: {
      create: [
        { title: 'Hello World', published: true },
        { title: 'Draft Post' }
      ]
    },
    profile: {
      create: { bio: 'I love coding!' }
    }
  },
  include: { posts: true, profile: true }
});

// Complex queries with filters
const publishedPosts = await prisma.post.findMany({
  where: {
    published: true,
    author: { email: { contains: '@company.com' } }
  },
  orderBy: { createdAt: 'desc' },
  take: 10,
  include: { author: { select: { name: true } } }
});

// Transactions
const [deletedPosts, deletedUser] = await prisma.$transaction([
  prisma.post.deleteMany({ where: { authorId: 1 } }),
  prisma.user.delete({ where: { id: 1 } })
]);

Drizzle — SQL-Like TypeScript ORM

schema.ts
import { pgTable, serial, text,
         timestamp, integer } from 'drizzle-orm/pg-core';

export const users = pgTable('users', {
  id: serial('id').primaryKey(),
  name: text('name').notNull(),
  email: text('email').unique().notNull(),
  createdAt: timestamp('created_at').defaultNow()
});

export const posts = pgTable('posts', {
  id: serial('id').primaryKey(),
  title: text('title').notNull(),
  content: text('content'),
  authorId: integer('author_id')
    .references(() => users.id)
});
queries.ts
import { db } from './db';
import { users, posts } from './schema';
import { eq, desc } from 'drizzle-orm';

// SQL-like syntax
const allUsers = await db
  .select()
  .from(users)
  .where(eq(users.name, 'Alice'));

// Joins feel natural
const usersWithPosts = await db
  .select({
    userName: users.name,
    postTitle: posts.title
  })
  .from(users)
  .leftJoin(posts, eq(users.id, posts.authorId))
  .orderBy(desc(posts.createdAt));
💡

Drizzle is lightweight (~7kb), has zero dependencies, and its queries look almost like SQL.

Common Database Patterns

Cache-Aside

Check Redis → Miss? → Query DB → Cache result

Reduce database load, improve response times

CQRS

Writes → PostgreSQL | Reads → Denormalized view

Separate read and write models for scale

Saga Pattern

Step1 → Step2 → Step3 | Fail? → Compensate

Distributed transactions across services

Event Sourcing

Store events → Rebuild state from history

Complete audit trail, time travel debugging

Database Performance Tips

01

Index Wisely

Index columns in WHERE, JOIN, ORDER BY. But indexes slow writes.

CREATE INDEX idx_users_email ON users(email);
02

Use EXPLAIN

Understand query execution plans before optimizing.

EXPLAIN ANALYZE SELECT * FROM users WHERE...
03

Avoid N+1 Queries

Fetch related data in one query, not loops.

include: { posts: true } // Prisma eager load
04

Connection Pooling

Reuse connections, don't create new ones per request.

new Pool({ max: 20 }) // pg pool
05

Paginate Results

Never SELECT * without LIMIT. Use cursor pagination for large sets.

LIMIT 20 OFFSET 0 // or cursor-based
06

Denormalize for Reads

Sometimes duplicating data beats expensive joins.

Store user_name on order, not just user_id

Database Security

CRITICAL Parameterized Queries

NEVER concatenate user input into SQL strings

HIGH Least Privilege

App user shouldn't have DROP TABLE permissions

HIGH Encrypt at Rest & Transit

TLS for connections, encryption for sensitive columns

MEDIUM Backup & Test Restores

Untested backups are not backups

MEDIUM Audit Logging

Track who changed what and when

2

Redis for speed, MongoDB for flexibility, Neo4j for graphs.

3
ORMs boost productivity

4
Security is non-negotiable

Parameterized queries. Always. No exceptions.

5
Understand before abstracting

Learn SQL even if you use an ORM. It will save you.
