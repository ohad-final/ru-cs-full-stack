Project Architecture

Structuring Scalable & Maintainable Applications

Why Project Architecture Matters

❌ Poor Architecture

  • Files scattered randomly
  • Circular dependencies
  • God objects & functions
  • "Where does this go?"
  • Onboarding takes weeks
  • Fear of refactoring

✓ Good Architecture

  • Logical grouping
  • Clear dependencies
  • Single responsibility
  • Obvious placement
  • Onboarding in days
  • Confident changes
"The goal of software architecture is to minimize the human resources required to build and maintain the required system." — Robert C. Martin

Benefits

  • Deletability — Remove feature = delete folder
  • Team ownership — Clear boundaries for teams

The Barrel Pattern (index.ts)

features/auth/index.ts

// Public API - what other features can import
export { LoginForm } from './components/LoginForm';
export { SignupForm } from './components/SignupForm';
export { useAuth } from './hooks/useAuth';
export type { User, AuthState } from './types';

// Internal components NOT exported
// PasswordStrength, FormField stay private

Importing from other features

// ✓ Clean import from public API
import { LoginForm, useAuth } from '@/features/auth';

// ✗ Don't reach into internals
import { PasswordStrength }
  from '@/features/auth/components/PasswordStrength';

Barrel files define your feature's public contract. Keep internals internal.

Part III

Layered Architecture

UI Layer Components, Pages, Styles
↓ ↑
Application Layer Hooks, State, Business Logic
↓ ↑
Domain Layer Entities, Rules, Types
↓ ↑
Infrastructure Layer API, Database, External Services

Rule: Dependencies flow downward. Lower layers never import from upper layers.

Clean Architecture in Practice

🎨 UI (React Components)

function UserProfile({ userId }) {
  const { user, loading } = useUser(userId);
  if (loading) return ;
  return ;
}

⚙️ Application (Hooks)

function useUser(userId: string) {
  const [user, setUser] = useState(null);
  useEffect(() => {
    userService.getById(userId)
      .then(setUser);
  }, [userId]);
  return { user, loading: !user };
}

🏢 Infrastructure (Services)

const userService = {
  async getById(id: string): Promise<User> {
    const res = await api.get(`/users/${id}`);
    return userMapper.toDomain(res.data);
  }
};

Dependency Injection

❌ Tight Coupling

// Hook directly imports API client
import { apiClient } from '@/lib/api';

function useUsers() {
  return useQuery({
    queryFn: () => apiClient.get('/users')
  });
}

// Can't test without real API!

✓ Dependency Injection

// Inject dependency via context
const ApiContext = createContext<ApiClient>();

function useUsers() {
  const api = useContext(ApiContext);
  return useQuery({
    queryFn: () => api.get('/users')
  });
}

// Easy to mock in tests!

Monorepo vs Multi-Repo

Multi-Repo

frontend
backend
shared-ui
mobile
  • Independent versioning
  • Clear ownership
  • Smaller CI/CD scope
  • Cross-repo changes painful
  • Dependency hell

Monorepo

apps/web
apps/api
packages/ui
packages/shared
  • Atomic cross-project changes
  • Shared tooling & config
  • Code reuse is trivial
  • Larger repo size
  • Needs good tooling

Part V

Configuration Management

🔐 Secrets (Never in code)

DATABASE_URL, API_KEYS, JWT_SECRET → Environment variables, Vault, AWS Secrets

🌍 Environment Config

API_URL, LOG_LEVEL, FEATURE_FLAGS → .env files, deployment config

⚙️ Application Config

PAGINATION_SIZE, RETRY_COUNT, TIMEOUTS → config/ directory in code

📝 Static Config

ROUTES, THEMES, CONSTANTS → constants/ directory in code

Environment Variables Best Practices

File Structure

.env.example      # Template (committed)
.env.local        # Local overrides (gitignored)
.env.development  # Dev defaults (committed)
.env.production   # Prod defaults (committed)
.env              # Runtime (gitignored)

Validation with Zod

import { z } from 'zod';

const envSchema = z.object({
  DATABASE_URL: z.string().url(),
  API_KEY: z.string().min(1),
  NODE_ENV: z.enum(['development', 'production']),
  PORT: z.coerce.number().default(3000),
});

export const env = envSchema.parse(process.env);

⚠️ Fail fast: Validate env vars at startup, not when first used.

Dependency Management

🎯 Be Intentional

Every dependency is a liability. Ask: Can I do this in 20 lines instead?

📊 Audit Regularly

npm audit
npx depcheck
npx npm-check-updates

🔒 Lock Versions

Commit package-lock.json. Use exact versions for critical deps.

📦 Organize by Type

{
  "dependencies": { /* Runtime */ },
  "devDependencies": { /* Build/test only */ },
  "peerDependencies": { /* Library consumers */ }
}

Part VII

SOLID in Frontend

S

Single Responsibility

One component = one job. Split UserDashboard into UserStats, UserActivity, UserSettings.

O

Open/Closed

Extend via props & composition, not modification. Use render props, children, slots.

L

Liskov Substitution

Components should be swappable. <TextInput> and <PasswordInput> share interface.

I

Interface Segregation

Don't force unused props. Split UserProps into UserDisplayProps & UserEditProps.

D

Dependency Inversion

Depend on abstractions. Inject API clients, don't import directly.

Component Composition

❌ Prop Drilling

<Card
  title="User"
  showAvatar={true}
  avatarSize="lg"
  avatarBorder={true}
  showBadge={true}
  badgeColor="green"
  // 20 more props...

✓ Composition

Part VIII

Code Documentation

✓ Document WHY, not WHAT

// Retry 3 times with exponential backoff
// because the payment API has occasional
// 502s during high traffic periods
const MAX_RETRIES = 3;

❌ Don't state the obvious

// Set the user name
const userName = user.name;

// Loop through users
users.forEach(user => {
  // Process each user
  processUser(user);
});

Code tells you what, comments tell you why. If you need to explain what, refactor the code.

Architecture Decision Records (ADRs)

# ADR-001: Use React Query for Server State

## Status
Accepted

## Context
We need to manage server state (API data) in our React app.
Options: Redux, Zustand, React Query, SWR.

## Decision
Use React Query because:
- Built-in caching and refetching
- Automatic background updates
- No boilerplate for common patterns
- Excellent DevTools

## Consequences
- Team needs to learn React Query patterns
- Existing Redux code will be migrated gradually
- Bundle size increases by ~12KB

ADRs capture the "why" behind decisions. Future you will thank present you.

Architecture Checklist

📁 Structure

  • Feature-based organization
  • Clear public APIs (barrel files)
  • Consistent naming conventions
  • Max 500 LOC per file

🔗 Dependencies

  • No circular imports
  • Dependencies flow downward
  • Injection over hard imports
  • Regular audits

⚙️ Configuration

  • Secrets in env vars only
  • Validated at startup
  • .env.example documented
  • Environment separation

📝 Documentation

  • README with quickstart
  • ADRs for decisions
  • Comments explain WHY
  • Types are documentation

Key Takeaways

1

Architecture is about change — Good architecture makes future changes easy, not impossible.

2

Colocate related code — Files that change together should live together.

3

Dependencies are liabilities — Every import is a coupling point. Be intentional.

4

Document decisions, not code — The code shows what, ADRs explain why.

5

Start simple, evolve — Don't over-engineer. Refactor as patterns emerge.
