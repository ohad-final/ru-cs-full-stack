The Scientific Method of Debugging

1

Observe

Gather information. What's the error? When does it happen? What changed?

→
2

Hypothesize

Form a theory. "The bug is probably in X because Y"

→
3

Experiment

Test your theory. Add logging, use debugger, write tests

→
4

Analyze

Did it confirm or refute? Update hypothesis, repeat

💡 Binary search: If you're stuck, comment out half the code. Does it still break?

Breakpoints & Debugging

🔴 Line Breakpoint

Click line number in Sources tab. Pause when line executes.

🟡 Conditional Breakpoint

Right-click line → "Add conditional". Only pause when condition is true.

user.id === 'abc123'

🟣 Logpoint

Log without modifying code. Right-click → "Add logpoint".

'User:', user.name, 'at', Date.now()

🔵 DOM Breakpoint

Pause when element changes. Right-click element → "Break on..."

Debugging React Renders

Common Render Issues

// ❌ Object created every render
<Button style={{ color: 'red' }} />

// ❌ Function created every render
<List onSelect={(item) => setSelected(item)} />

// ❌ Array filter every render
<Items data={items.filter(i => i.active)} />

Solutions

// ✓ Memoize objects
const style = useMemo(() => ({ color: 'red' }), []);

// ✓ useCallback for functions
const onSelect = useCallback((item) => {
  setSelected(item);
}, []);

// ✓ useMemo for derived data
const activeItems = useMemo(() =>
  items.filter(i => i.active), [items]);

Enable "Highlight updates" in React DevTools to visualize re-renders

Debugging Async Code

The Challenge

// Stack trace doesn't show origin
async function fetchUser() {
  const response = await fetch('/api/user');
  const data = await response.json();  // Error here!
  return data;
}

// Who called this? Stack trace is useless

Solutions

// 1. Enable async stack traces
// In Chrome: DevTools Settings →
//   "Async stack traces"

// 2. Add context to errors
async function fetchUser() {
  try {
    const res = await fetch('/api/user');
    return await res.json();
  } catch (error) {
    error.context = { endpoint: '/api/user' };
    throw error;
  }
}
Part V

Structured Logging

❌ Unstructured

console.log('User ' + userId + ' failed login');
console.log('Error: ' + error.message);
// Hard to search, parse, or analyze

✓ Structured (JSON)

logger.info('Login attempt failed', {
  userId,
  reason: 'invalid_password',
  ip: req.ip,
  timestamp: Date.now(),
  traceId: req.traceId
});

// Easy to search: userId=abc123
// Easy to aggregate: count by reason

Popular Libraries

pino winston bunyan

Log Levels

ERROR Something broke. Action required. logger.error('Payment failed', { orderId, error })
WARN Something unexpected. Monitor it. logger.warn('Rate limit approaching', { usage: 90 })
INFO Normal operations. Business events. logger.info('Order completed', { orderId, total })
DEBUG Detailed info for debugging. logger.debug('Cache hit', { key, ttl })

Production: INFO and above. Development: DEBUG and above.

Correlation IDs

// Middleware to add trace ID to every request
app.use((req, res, next) => {
  req.traceId = req.headers['x-trace-id'] || crypto.randomUUID();
  res.setHeader('x-trace-id', req.traceId);
  next();
});

// Include in every log
app.get('/api/orders', async (req, res) => {
  logger.info('Fetching orders', { traceId: req.traceId });

  const orders = await db.orders.findMany();
  logger.info('Orders found', {
    traceId: req.traceId,
    count: orders.length
  });

  res.json(orders);
});

Now you can trace a single request across all services and logs!

Part VI

Error Tracking with Sentry

Frontend Setup

import * as Sentry from '@sentry/react';

Sentry.init({
  dsn: 'https://xxx@sentry.io/123',
  environment: process.env.NODE_ENV,
  release: process.env.VERSION,
  integrations: [
    Sentry.browserTracingIntegration(),
    Sentry.replayIntegration(),
  ],
  tracesSampleRate: 0.1,  // 10% of transactions
  replaysOnErrorSampleRate: 1.0,  // 100% on error
});

What You Get

  • Automatic error capture
  • Source maps for stack traces
  • User context & breadcrumbs
  • Session replay (video!)
  • Performance monitoring
  • Release tracking

Error Boundaries in React

import * as Sentry from '@sentry/react';

function App() {
  return (
    <Sentry.ErrorBoundary
      fallback={({ error, resetError }) => (
        <div className="error-page">
          <h1>Something went wrong</h1>
          <p>{error.message}</p>
          <button onClick={resetError}>Try again</button>
        </div>
      )}
      onError={(error, componentStack) => {
        // Already sent to Sentry automatically
        console.error('React error:', error);
      }}
    >
      <Router>
        <Routes />
      </Router>
    </Sentry.ErrorBoundary>
  );
}

Wrap at multiple levels: App, Route, Feature — prevent one bug from crashing everything

Custom Error Context

// Add user context
Sentry.setUser({
  id: user.id,
  email: user.email,
  subscription: user.plan,
});

// Add breadcrumbs (what led to error)
Sentry.addBreadcrumb({
  category: 'ui.click',
  message: 'User clicked checkout button',
  level: 'info',
});

// Capture with extra context
try {
  await processPayment(order);
} catch (error) {
  Sentry.captureException(error, {
    extra: {
      orderId: order.id,
      amount: order.total,
      paymentMethod: order.payment.type,
    },
    tags: {
      feature: 'checkout',
      priority: 'high',
    },
  });
}

Part IX

Debugging in Production

⚠️ Ground Rules

  • Never attach debugger to production
  • Don't add console.log and redeploy
  • Use structured logging you added beforehand
  • Reproduce in staging when possible

Safe Production Tools

  • Log aggregation — Search across all instances
  • Distributed tracing — Follow request path
  • Error tracking — Sentry, Bugsnag
  • Feature flags — Disable problematic code
  • Database query logs — Slow query analysis
