GitHub Actions: Your First Workflow

.github/workflows/ci.yml

name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - run: npm ci
      - run: npm run lint
      - run: npm run test
      - run: npm run build

This runs on every push and PR to main. Tests must pass to merge.

GitHub Actions: Key Concepts

🎯 Triggers (on:)

on:
  push:          # On push to branches
  pull_request:  # On PR events
  schedule:      # Cron schedule
  workflow_dispatch: # Manual trigger

💼 Jobs

jobs:
  build:
    runs-on: ubuntu-latest
  test:
    needs: build  # Depends on build
    runs-on: ubuntu-latest

📦 Steps & Actions

steps:
  - uses: actions/checkout@v4
  - run: npm install
  - uses: my-org/custom-action@v1

🔐 Secrets

env:
  DATABASE_URL: ${{ secrets.DATABASE_URL }}
  API_KEY: ${{ secrets.API_KEY }}

Matrix Builds

jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, macos-latest, windows-latest]
        node: [18, 20, 22]

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm ci
      - run: npm test

Deploy to Vercel with GitHub Actions

name: Deploy

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Deploy to Vercel
        uses: amondnet/vercel-action@v25
        with:
          vercel-token: ${{ secrets.VERCEL_TOKEN }}
          vercel-org-id: ${{ secrets.ORG_ID }}
          vercel-project-id: ${{ secrets.PROJECT_ID }}
          vercel-args: '--prod'

💡 Easier way: Just connect your GitHub repo to Vercel — automatic deploys on every push!

Why Docker?

The Problem

Docker's Solution

App Node 20 Dependencies Config

Docker Concepts

📄 Dockerfile

Recipe to build an image. Instructions for what goes in the container.

📦 Image

Immutable snapshot. Like a class definition — template for containers.

🏃 Container

Running instance of an image. Like an object — live process with state.

🗄️ Registry

Storage for images. Docker Hub, GitHub Container Registry, AWS ECR.

Your First Dockerfile

# Use official Node.js image as base
FROM node:20-alpine

# Set working directory
WORKDIR /app

# Copy package files first (better caching)
COPY package*.json ./

# Install dependencies
RUN npm ci --only=production

# Copy source code
COPY . .

# Build the application
RUN npm run build

# Expose the port
EXPOSE 3000

# Start the application
CMD ["npm", "start"]
docker build -t my-app . docker run -p 3000:3000 my-app

Multi-Stage Builds

# Stage 1: Build
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Stage 2: Production
FROM node:20-alpine AS runner
WORKDIR /app

# Copy only what we need
COPY --from=builder /app/package*.json ./
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/dist ./dist

# Don't run as root
USER node

EXPOSE 3000
CMD ["node", "dist/index.js"]
1.2 GB → 150 MB

Final image only contains production artifacts

⚠️ Golden Rule: Staging should mirror production as closely as possible. Same infra, same config, same data shape.

Branch-Based Deployments

name: Deploy

on:
  push:
    branches: [main, staging, develop]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set environment
        run: |
          if [ "${{ github.ref }}" = "refs/heads/main" ]; then
            echo "ENV=production" >> $GITHUB_ENV
          elif [ "${{ github.ref }}" = "refs/heads/staging" ]; then
            echo "ENV=staging" >> $GITHUB_ENV
          else
            echo "ENV=development" >> $GITHUB_ENV
          fi

      - name: Deploy to ${{ env.ENV }}
        run: ./deploy.sh ${{ env.ENV }}
main → production staging → staging develop → development

Infrastructure as Code (IaC)

❌ Manual Setup

  • Click through AWS console
  • SSH and run commands
  • "I think I changed something..."
  • Impossible to reproduce
  • No audit trail

✓ Infrastructure as Code

  • Define infra in files
  • Version control everything
  • Review changes in PRs
  • Reproducible environments
  • Full audit history

Deploy code but control visibility. Decouple deployment from release.

  const showNewUI = useFeatureFlag('new-dashboard-ui');

  if (showNewUI) {

  }

  return ;
}

Rollback Strategies

🏷️ Revert to Previous Version

# Kubernetes
kubectl rollout undo deployment/my-app

# Vercel
vercel rollback

# Docker
docker run my-app:v1.2.3

🚦 Feature Flag Kill Switch

// Instantly disable feature
await flags.disable('new-checkout-flow');

// No deployment needed!

🗃️ Database Rollback

# Revert migration
npx prisma migrate resolve --rolled-back

# Restore from backup
pg_restore -d mydb backup.dump

⚠️ Always test rollback procedures before you need them!

Part VIII

The Three Pillars of Observability

📊 Metrics

Numerical measurements over time

  • Request rate
  • Error rate
  • Response time (p50, p95, p99)
  • CPU/Memory usage
Prometheus, Datadog, CloudWatch

📝 Logs

Discrete events with context

  • Error messages
  • Request details
  • User actions
  • System events
ELK Stack, Loki, CloudWatch Logs

🔍 Traces

Request flow across services

  • Service dependencies
  • Latency breakdown
  • Error propagation
  • Bottleneck identification
Jaeger, Zipkin, Honeycomb

Alerting Best Practices

✓ Good Alerts

  • Error rate > 1% for 5 minutes
  • p99 latency > 2s for 10 minutes
  • Database connections > 80%
  • Payment failures spike

❌ Alert Fatigue

  • CPU briefly spiked to 70%
  • Single 500 error occurred
  • Disk at 50% (non-urgent)
  • Too many channels

Severity Levels

🚨 Critical: Page immediately ⚠️ Warning: Check within hours ℹ️ Info: Review in standup

DevOps Checklist

🔄 CI/CD

  • Automated builds on every push
  • Tests run in pipeline
  • Linting enforced
  • Deploy on merge to main

📦 Containers

  • Dockerfile for all services
  • Multi-stage builds
  • Docker Compose for local dev
  • Images scanned for vulnerabilities

🔐 Security

  • No secrets in code
  • Dependency scanning
  • Secret rotation policy
  • Least privilege access

📊 Monitoring

  • Health check endpoints
  • Structured logging
  • Key metrics dashboards
  • Actionable alerts only

Key Takeaways

1

Automate everything — If you do it twice, script it. If you script it, put it in CI.

2

Small, frequent deploys — Deploy daily, not monthly. Smaller changes = smaller risks.

3

Infrastructure is code — Version control your infra. Review changes in PRs.

4

Shift left on security — Find vulnerabilities in CI, not production.

5

Measure what matters — Good observability makes debugging 10x faster.
