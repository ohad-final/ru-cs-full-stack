SQL Injection — The Classic

VULNERABLE CODE
// User input directly in SQL string
const query = `
    SELECT * FROM users
    WHERE username = '${username}'
    AND password = '${password}'
`;

// Attacker enters:
// username: admin' --
// password: anything

// Resulting query:
SELECT * FROM users
WHERE username = 'admin' --'
AND password = 'anything'

// The -- comments out the rest!
// Logs in as admin without password!
SECURE CODE
// Parameterized query (prepared statement)
const query = `
    SELECT * FROM users
    WHERE username = $1
    AND password = $2
`;

const result = await db.query(query, [
    username,
    password
]);

// The database treats $1 and $2 as DATA
// not as SQL code — injection impossible!
Rule #1: NEVER concatenate user input into SQL strings. Always use parameterized queries.

NoSQL Injection — MongoDB Edition

VULNERABLE
// Express route with JSON body
app.post('/login', async (req, res) => {
    const { username, password } = req.body;

    const user = await db.collection('users').findOne({
        username: username,
        password: password
    });
});

// Attacker sends:
{
    "username": "admin",
    "password": { "$ne": "" }
}

// Query becomes:
// Find user where username = "admin"
// AND password != ""
// Bypasses authentication!
SECURE
// Validate input types!
app.post('/login', async (req, res) => {
    const { username, password } = req.body;

    // Ensure strings, not objects
    if (typeof username !== 'string' ||
        typeof password !== 'string') {
        return res.status(400).json({
            error: 'Invalid input'
        });
    }

    // Now safe to query
    const user = await db.collection('users').findOne({
        username,
        password: await hash(password)
    });
});

Command Injection — Shell Shock

VULNERABLE
// Image resize endpoint
app.post('/resize', (req, res) => {
    const { filename, width } = req.body;

    // DANGER: User input in shell command!
    exec(`convert ${filename} -resize ${width} output.jpg`);
});

// Attacker sends:
{
    "filename": "image.jpg; rm -rf /",
    "width": "100"
}

// Executed command:
// convert image.jpg; rm -rf / -resize 100 output.jpg
//                    ^^^^^^^^^ DELETES EVERYTHING
SECURE
import { execFile } from 'child_process';

app.post('/resize', (req, res) => {
    const { filename, width } = req.body;

    // Validate filename (whitelist chars)
    if (!/^[a-zA-Z0-9_.-]+$/.test(filename)) {
        return res.status(400).json({
            error: 'Invalid filename'
        });
    }

    // execFile doesn't use shell
    // Arguments are passed directly
    execFile('convert', [
        filename,
        '-resize', String(width),
        'output.jpg'
    ]);
});

XSS Prevention

VULNERABLE
// React: dangerouslySetInnerHTML
<div dangerouslySetInnerHTML={{
    __html: userInput  // XSS!
}} />

// Vanilla JS: innerHTML
element.innerHTML = userInput;  // XSS!

// jQuery: html()
$('#output').html(userInput);  // XSS!

// Template literal in HTML
const html = `<p>${userInput}</p>`;  // XSS!

// URL without validation
<a href={userUrl}>Click</a>
// javascript:alert(1) works!
SECURE
// React: default escaping (safe!)
<div>{userInput}</div>

// Vanilla JS: textContent
element.textContent = userInput;  // Safe!

// jQuery: text()
$('#output').text(userInput);  // Safe!

// Sanitize if HTML needed
import DOMPurify from 'dompurify';
const clean = DOMPurify.sanitize(userInput);

// Validate URLs
const url = new URL(userUrl);
if (!['http:', 'https:'].includes(url.protocol)) {
    throw new Error('Invalid URL');
}

Content Security Policy — XSS Killswitch

HTTP Header
Content-Security-Policy:
    default-src 'self';
    script-src 'self' https://trusted-cdn.com;
    style-src 'self' 'unsafe-inline';
    img-src 'self' data: https:;
    connect-src 'self' https://api.example.com;
    frame-ancestors 'none';
    form-action 'self';
script-src 'self'

Only scripts from same origin — blocks inline scripts!

Only fetch/XHR to these domains

frame-ancestors 'none'

Prevents clickjacking — can't be embedded in iframe

Authentication Disasters

How companies lose millions of passwords

password = "123456"  // Still #1 most common password

Password Storage — Hall of Shame

WORST

Plain Text

password = "hunter2"

RockYou (2009): 32M passwords in plain text

BAD

MD5 Hash

MD5("hunter2") = "2ab96390...

LinkedIn (2012): MD5 without salt, cracked in hours

BAD

SHA-1 Unsalted

SHA1("hunter2") = "a8a00add..."

Adobe (2013): 150M accounts, same password = same hash

GOOD

bcrypt/Argon2

$2b$12$LQv3c1yqBWVHxkd0LHAkCOYz6...

Slow, salted, designed for passwords

Proper Password Hashing

WRONG
// Plain text (NEVER!)
db.insert({ password: userPassword });

// Simple hash (BROKEN!)
const hash = crypto
    .createHash('sha256')
    .update(password)
    .digest('hex');

// Problems:
// 1. Rainbow tables crack common passwords
// 2. Same password = same hash
// 3. Too fast — billions of guesses/sec
// 4. No defense against GPU cracking
CORRECT
import bcrypt from 'bcrypt';

// Registration
const saltRounds = 12;  // Cost factor
const hash = await bcrypt.hash(password, saltRounds);
db.insert({ passwordHash: hash });

// Login
const isValid = await bcrypt.compare(
    inputPassword,
    user.passwordHash
);

// Why bcrypt works:
// 1. Random salt per password
// 2. Intentionally slow (~100ms)
// 3. Cost factor increases over time
// 4. Resistant to GPU attacks

Session Security

Session Hijacking

Attacker steals session cookie -> Impersonates user
Defense: HttpOnly, Secure flags, short expiry

Session Fixation

Attacker sets victim's session ID -> Victim logs in -> Attacker uses same session
Defense: Regenerate session ID on login
Secure Cookie Settings
res.cookie('session', sessionId, {
    httpOnly: true,    // No JavaScript access
    secure: true,      // HTTPS only
    sameSite: 'strict', // No cross-site sending
    maxAge: 3600000    // 1 hour expiry
});

Broken Access Control

When "just change the ID" works

/api/users/123/documents  ->  /api/users/124/documents

IDOR — Insecure Direct Object Reference

VULNERABLE
// API endpoint
app.get('/api/invoices/:id', async (req, res) => {
    const invoice = await Invoice.findById(req.params.id);
    res.json(invoice);
});

// User requests their invoice
GET /api/invoices/1001

// Attacker changes ID
GET /api/invoices/1002  // Someone else's!
GET /api/invoices/1003  // Another one!

// No authorization check = data breach
Real IDOR Bugs
  • Panera Bread: 37M customer records
  • US Army: Soldier medical records
  • Instagram: Any account photos
SECURE
// Always verify ownership!
app.get('/api/invoices/:id', async (req, res) => {
    const invoice = await Invoice.findOne({
        _id: req.params.id,
        userId: req.user.id  // Must belong to user!
    });

    if (!invoice) {
        return res.status(404).json({
            error: 'Invoice not found'
        });
    }

    res.json(invoice);
});

// Or use UUIDs instead of sequential IDs
// Harder to guess: /api/invoices/f47ac10b-58cc-...

Privilege Escalation

Horizontal Escalation

User A -> User B's data

Input Validation — Trust Nothing

1. Validate on Server

Client-side validation is for UX. Server-side is for security.

2. Whitelist, Don't Blacklist

Define what's allowed, reject everything else.

3. Validate Type, Length, Format, Range

email: string, max 254 chars, valid format

Zod Validation Example
import { z } from 'zod';

const UserSchema = z.object({
    email: z.string().email().max(254),
    password: z.string().min(8).max(100),
    age: z.number().int().min(13).max(120),
    username: z.string().regex(/^[a-zA-Z0-9_]+$/).max(30),
});

// In your route
const result = UserSchema.safeParse(req.body);
if (!result.success) {
    return res.status(400).json({ errors: result.error.issues });
}

Key Takeaways

01
Never trust user input

Validate everything. Use parameterized queries. Encode output.

02
Authentication is hard — use proven libraries

bcrypt for passwords, established session libraries, MFA when possible.

03
Defense in depth

Multiple layers: validation, authentication, authorization, encryption, monitoring.

04
Security is a process, not a product

Keep dependencies updated. Monitor logs. Have an incident response plan.
