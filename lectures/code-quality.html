Traditional advice: Many unit tests, fewer integration tests, even fewer E2E tests.

But is this still the best approach?

The Testing Trophy

Kent C. Dodds' alternative — more integration tests

E2E
Integration
Unit
Static (TypeScript, ESLint)
"Write tests. Not too many. Mostly integration."
— Guillermo Rauch (Vercel CEO)

Why more integration tests?

  • Test components as users use them
  • Higher confidence than isolated unit tests
  • Less brittle than E2E tests
  • Modern tooling (RTL, MSW) makes them fast

What to Test

Must Test

  • Authentication flows
  • Payment processing
  • Core business logic
  • Data validation
  • Error handling

Should Test

  • User journeys (happy path)
  • Edge cases you've found
  • Previously broken features
  • Complex calculations

Maybe Skip

  • Simple getters/setters
  • Framework code (React, Next.js)
  • Third-party libraries
  • Trivial UI styling
Regression test: When you fix a bug, write a test that would have caught it.
Part II

Unit Testing with Jest

Fast, isolated tests for functions and modules

Jest Basics

sum.test.ts
import { sum, multiply } from './math';

describe('Math utilities', () => {
    describe('sum', () => {
        it('adds two positive numbers', () => {
            expect(sum(2, 3)).toBe(5);
        });

        it('handles negative numbers', () => {
            expect(sum(-1, 5)).toBe(4);
        });

        it('returns 0 for empty input', () => {
            expect(sum()).toBe(0);
        });
    });

    describe('multiply', () => {
        it('multiplies two numbers', () => {
            expect(multiply(3, 4)).toBe(12);
        });
    });
});
npm test — Run all tests npm test -- --watch — Watch mode npm test -- sum.test.ts — Run specific file

Mocking in Jest

Mock Functions
// Create a mock function
const mockFn = jest.fn();
mockFn('arg1', 'arg2');

expect(mockFn).toHaveBeenCalled();
expect(mockFn).toHaveBeenCalledWith('arg1', 'arg2');
expect(mockFn).toHaveBeenCalledTimes(1);

// Mock return value
mockFn.mockReturnValue(42);
mockFn.mockResolvedValue({ data: 'async' });
Mock Modules
// Mock an entire module
jest.mock('./api', () => ({
    fetchUser: jest.fn().mockResolvedValue({
        id: 1,
        name: 'Test User'
    })
}));

// Mock only specific exports
jest.mock('./utils', () => ({
    ...jest.requireActual('./utils'),
    sendEmail: jest.fn()
}));
Warning: Over-mocking leads to tests that pass but don't catch real bugs.

Testing Async Code

async.test.ts
// Using async/await (recommended)
it('fetches user data', async () => {
    const user = await fetchUser(1);
    expect(user.name).toBe('John');
});

// Testing rejected promises
it('handles errors', async () => {
    await expect(fetchUser(-1)).rejects.toThrow('Invalid ID');
});

// Testing with timers
it('debounces input', () => {
    jest.useFakeTimers();

    const callback = jest.fn();
    const debounced = debounce(callback, 500);

    debounced();
    debounced();
    debounced();

    expect(callback).not.toHaveBeenCalled();

    jest.advanceTimersByTime(500);

    expect(callback).toHaveBeenCalledTimes(1);
});

Coverage Lies

  • 100% coverage ≠ bug-free code
  • High coverage with bad assertions = false confidence
  • Sensible target: 70-80% for most projects

Part III

Component Testing

React Testing Library — Test behavior, not implementation

React Testing Library Philosophy

"The more your tests resemble the way your software is used, the more confidence they can give you."
— Testing Library Guiding Principle

Don't Test Implementation

// Testing internal state
expect(component.state.isOpen).toBe(true);

// Testing CSS classes
expect(button).toHaveClass('btn-active');

// Testing component instances
expect(wrapper.instance().handleClick).toBeCalled();

Test User Behavior

// Test what user sees
expect(screen.getByText('Menu is open')).toBeVisible();

// Test accessibility
expect(screen.getByRole('button')).toBeEnabled();

// Test user interactions
await userEvent.click(button);
expect(screen.getByText('Submitted!')).toBeInTheDocument();

RTL Queries

getBy*

Throws if not found. Use when element should exist.

screen.getByRole('button', { name: 'Submit' })

queryBy*

Returns null if not found. Use to assert absence.

expect(screen.queryByText('Error')).not.toBeInTheDocument()

findBy*

Returns promise. Use for async elements.

await screen.findByText('Loaded!')

Query Priority (Best to Worst)

  1. getByRole — Accessible to everyone
  2. getByLabelText — Form inputs
  3. getByPlaceholderText — If no label
  4. getByText — Non-interactive elements
  5. getByTestId — Last resort

Testing User Interactions

LoginForm.test.tsx
import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import { LoginForm } from './LoginForm';

it('submits form with user credentials', async () => {
    const user = userEvent.setup();
    const onSubmit = jest.fn();

    render(<LoginForm onSubmit={onSubmit} />);

    // Type in form fields
    await user.type(screen.getByLabelText('Email'), 'test@example.com');
    await user.type(screen.getByLabelText('Password'), 'secret123');

    // Click submit button
    await user.click(screen.getByRole('button', { name: 'Log In' }));

    // Assert submission
    expect(onSubmit).toHaveBeenCalledWith({
        email: 'test@example.com',
        password: 'secret123'
    });
});

Mocking APIs with MSW

Mock Service Worker — Intercept requests at the network level

handlers.ts
import { http, HttpResponse } from 'msw';

export const handlers = [
    http.get('/api/user/:id', ({ params }) => {
        return HttpResponse.json({
            id: params.id,
            name: 'Test User',
            email: 'test@example.com'
        });
    }),

    http.post('/api/login', async ({ request }) => {
        const { email, password } = await request.json();

        if (password === 'wrong') {
            return HttpResponse.json(
                { error: 'Invalid credentials' },
                { status: 401 }
            );
        }

        return HttpResponse.json({ token: 'fake-jwt' });
    })
];
Why MSW? Real fetch/axios calls, works in tests and browser, no implementation mocking.
Part IV

E2E Testing with Playwright

Real browsers, full stack confidence

Why E2E Tests?

🌐

Real Browser

Chrome, Firefox, Safari — actual rendering

📚

Full Stack

Frontend + Backend + Database together

👤

User Perspective

Test complete user journeys

✅

Highest Confidence

If E2E passes, it really works

Tradeoffs

  • Slow — Seconds per test vs milliseconds
  • Flaky — Network, timing, environment issues
  • Expensive — CI minutes, maintenance burden

Rule: Use E2E for critical paths only. Don't test everything E2E.

Playwright Basics

login.spec.ts
import { test, expect } from '@playwright/test';

test('user can log in', async ({ page }) => {
    // Navigate to login page
    await page.goto('/login');

    // Fill in credentials
    await page.getByLabel('Email').fill('user@example.com');
    await page.getByLabel('Password').fill('password123');

    // Click login button
    await page.getByRole('button', { name: 'Log In' }).click();

    // Assert successful login
    await expect(page).toHaveURL('/dashboard');
    await expect(page.getByText('Welcome back!')).toBeVisible();
});

test('shows error for invalid credentials', async ({ page }) => {
    await page.goto('/login');
    await page.getByLabel('Email').fill('wrong@example.com');
    await page.getByLabel('Password').fill('wrongpass');
    await page.getByRole('button', { name: 'Log In' }).click();

    await expect(page.getByText('Invalid credentials')).toBeVisible();
});

Writing Resilient E2E Tests

Flaky
// Hard-coded waits
await page.waitForTimeout(3000);

// Fragile selectors
await page.click('.btn-primary');
await page.click('#submit-form > div:nth-child(2) > button');

// Race conditions
await page.click('button');
const text = await page.textContent('.result');
Resilient
// Wait for specific state
await page.waitForSelector('[data-loaded="true"]');

// Accessible selectors
await page.getByRole('button', { name: 'Submit' }).click();

// Auto-waiting assertions
await expect(page.getByText('Success')).toBeVisible();

// Network idle
await page.waitForLoadState('networkidle');
Playwright auto-waits for elements to be actionable. Trust it. Don't add manual waits.

Visual Testing

visual.spec.ts
import { test, expect } from '@playwright/test';

test('homepage looks correct', async ({ page }) => {
    await page.goto('/');

    // Full page screenshot comparison
    await expect(page).toHaveScreenshot('homepage.png');
});

test('button states', async ({ page }) => {
    await page.goto('/components');

    // Component screenshot
    const button = page.getByRole('button', { name: 'Submit' });
    await expect(button).toHaveScreenshot('button-default.png');

    await button.hover();
    await expect(button).toHaveScreenshot('button-hover.png');
});

Subsequent runs: Compares against baseline, fails on diff

Linting & Formatting

Automated code quality enforcement

Why Lint Your Code?

🐛

Catch Bugs Early

Unused variables, unreachable code, type errors

📏

Enforce Consistency

Same style across entire codebase

🤝

End Style Debates

Tabs vs spaces? The linter decides.

🚀

New devs learn conventions from errors

"Code is read more than it is written. Consistency reduces cognitive load."

ESLint Configuration

eslint.config.js (Flat Config)
import js from '@eslint/js';
import typescript from '@typescript-eslint/eslint-plugin';
import react from 'eslint-plugin-react';
import reactHooks from 'eslint-plugin-react-hooks';

export default [
    js.configs.recommended,
    {
        files: ['**/*.{ts,tsx}'],
        plugins: { typescript, react, reactHooks },
        rules: {
            'no-unused-vars': 'error',
            'no-console': 'warn',
            'react-hooks/rules-of-hooks': 'error',
            'react-hooks/exhaustive-deps': 'warn',
            '@typescript-eslint/no-explicit-any': 'error',
        }
    }
];

Pre-Commit Hooks

Automatically run checks before every commit

Setup
# Install
npm install -D husky lint-staged

# Initialize husky
npx husky init

# Add pre-commit hook
echo "npx lint-staged" > .husky/pre-commit

{
    "lint-staged": {
        "*.{ts,tsx}": ["eslint --fix", "prettier --write"],
        "*.{json,md}": ["prettier --write"]

Branching Strategies

GitHub Flow (Recommended)

main feature-branch → PR → merge

Simple. One main branch, feature branches, PRs to merge.

Best for: Most web apps, continuous deployment

GitFlow

main develop feature/* release/*

Complex. Multiple long-lived branches, release cycles.

Best for: Versioned releases, mobile apps

Trunk-Based

main (everyone commits here)

Extreme simplicity. Feature flags instead of branches.

Best for: High-trust teams, Google-style CI

Feature Branch Workflow

1
git checkout -b feature/add-login

Create and switch to feature branch

2
git add . && git commit -m "feat: add login form"

Make commits (small, focused)

3
git push -u origin feature/add-login

Push to remote

4
Open Pull Request on GitHub

Request code review

5
Merge PR after approval

Squash or merge commit

6
git checkout main && git pull

Update local main

Writing Good Pull Requests

Title

feat: Add user authentication with JWT Fixed stuff

Description

  • What: Brief summary of changes
  • Why: Motivation, linked issue
  • How: Implementation approach (if complex)
  • Testing: How to verify it works
  • Screenshots: For UI changes

Best Practices

  • Small PRs — Easier to review, faster to merge
  • One concern — Don't mix refactoring with features
  • Self-review first — Catch obvious issues yourself
  • Respond to feedback — Discuss, don't argue

Merge vs Rebase

git merge

main:    A---B---C---M
                     \     /
feature:  D---E---
  • Preserves complete history
  • Creates merge commit (M)
  • Non-destructive
Use for: Public branches, PRs, team work

git rebase

main:    A---B---C
                         \
feature:          D'---E'
  • Linear history
  • Rewrites commits (D→D')
  • Cleaner git log
Use for: Local branches before pushing, updating feature from main
Golden Rule: Never rebase public/shared branches. Only rebase your own unpushed work.

Never Commit

  • API keys & tokens
  • Database passwords
  • .env files with secrets
  • Private keys
  • OAuth client secrets

If You Accidentally Commit Secrets

  1. Revoke/rotate the secret immediately
  2. Use git filter-branch or BFG Repo-Cleaner
  3. Force push (coordinate with team)

Pre-PR Quality Checklist

Before Committing

  • [ ] Code compiles/builds without errors
  • [ ] All tests pass locally
  • [ ] No linting errors
  • [ ] No console.log() statements
  • [ ] No commented-out code

Before Opening PR

  • [ ] Branch is up-to-date with main
  • [ ] Self-reviewed the diff
  • [ ] Tests added for new functionality
  • [ ] PR description is complete
  • [ ] Screenshots for UI changes

Key Takeaways

01
Write tests for confidence, not coverage

Test critical paths and behavior. 80% coverage with good tests > 100% with bad ones.

02
Automate quality enforcement

Linting + pre-commit hooks = no style debates, consistent code.

03
Small PRs, frequent commits

Easier to review, easier to revert, faster feedback.

04
Git is your safety net

Commit often, branch freely. You can always go back.
