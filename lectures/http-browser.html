HTTP/3 — How Does It Start?

1
First visit → HTTP/2 over TCP

Browser connects normally. Server responds with:

Alt-Svc: h3=":443"; ma=86400

"I support HTTP/3 on port 443, remember for 24h"

2
Browser tries QUIC in background

Sends UDP probe while TCP connection continues

If QUIC works → switch. If blocked → stay on TCP.

3
Next visit → Direct HTTP/3

Browser remembers Alt-Svc, connects via QUIC immediately

0-RTT: Send request WITH connection setup

Why HTTP/3 Might Not Work

🔥 Firewall blocks UDP 443 🏢 Corporate proxy only allows TCP 📡 Unstable UDP on some networks ⏳ Server hasn't enabled QUIC yet

When Should You Use It?

Great for: Mobile users, high-latency, lossy networks, global CDN
Less benefit: Low-latency LANs, single requests, already fast sites

Use a CDN like Cloudflare — they handle HTTP/3 automatically with TCP fallback.

What Could Be Done Differently?

🤔 Stateful by Design?

Built-in session management instead of cookie hacks. But: scalability nightmare, lose REST simplicity.

📡 Push-First Model?

WebSocket-style bi-directional from day one. But: resource management complexity, harder caching.

🔐 Mandatory Encryption?

HTTPS-only from the start. We're almost there now, but 30 years of mixed content headaches.

📦 Built-in Compression?

Automatic gzip/brotli at protocol level. Content-Encoding is opt-in and often misconfigured.

🎯 Native Streaming?

First-class support for real-time data. SSE and WebSockets are afterthoughts.

🔄 Versioning in URL?

REST debates: /v1/api vs Accept header. Neither is "correct" — protocol gives no guidance.
